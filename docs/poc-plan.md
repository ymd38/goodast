## 0. このPoCで「検証したいこと」（最優先）

PoCの目的は機能網羅ではなく、**最大の技術リスクを縦串で潰すこと**。以下が動けば成功とする。

1. **エンジン統合リスク**: Nuclei v3 Go SDK をワーカープロセスに埋め込み、UIから渡したURLをスキャンして結果をコールバックで受け取りDBへ保存できる。
2. **非同期ジョブリスク**: スキャンは数分かかる前提で、API／ワーカーを分離しジョブキューで非同期実行できる。
3. **認証後スキャンリスク**: ログイン済みセッション（Cookie / Bearer）を持ち込んで、認証後ページを安全にスキャンできる。
4. **状態・遷移の可視化リスク（PoCの主役）**: サイト名をキーに診断を時系列で蓄積し、「直近の検査状態」と「時系列の遷移（改善/悪化）」をツール内ダッシュボードで可視化できる。
5. **AI連携リスク**: BYOK（ユーザー持ち込みキー）でLLMを呼び、検出結果を初心者向けに平易化できる。

> この5本が通れば、以降の機能拡張は「足し算」になる。逆にここでブロックされる要素を先に発見するのがPoCの価値。

## 2. 検査可能な脆弱性の範囲

DASTで動的に確実に取れるカテゴリに絞る。Nucleiの `-dast`（ファジング）テンプレート + 既存テンプレートでカバー。

### 2.1 PoC対象カテゴリ
| カテゴリ | OWASP | 備考 |
|---|---|---|
| インジェクション（SQLi/コマンド等） | A03 | `-dast` ファジング |
| XSS（反射型・格納型） | A03 | ファジング |
| SSRF | A10 | Interactsh/OASTによる帯域外検知 |
| セキュリティ設定不備（ヘッダ不足・verboseエラー等） | A05 | DASTが最も得意 |
| TLS/暗号設定（一部） | A02 | `ssl` プロトコルテンプレート |
| オープンリダイレクト / CRLF | A01他 | ファジング |
| 既知の脆弱コンポーネント/CVE | A06 | テンプレート方式 |

### 2.2 認証後ページの扱い（要件: ログイン後を範囲に含める）
- **PoC方式 = セッション持ち込み（Session Injection）**:
  - ユーザーがUIで「ログイン済みCookie」または「Authorizationヘッダ（Bearer等）」を登録。
  - スキャン時、Nucleiの全リクエストに当該ヘッダ／Cookieを付与（SDKの `WithHeaders` 相当）。
  - 「ブラウザでログイン → DevToolsからCookieをコピーして貼る」手順をUIでガイド（初心者向けに図解）。
- **なぜこの方式か**: フォームログイン自動化（Nuclei secrets.yaml / ZAP認証コンテキスト）は設定が複雑で、PoCのリスク検証には過剰。セッション持ち込みは確実で説明しやすく、認証後スキャンの成立を最小コストで検証できる。
- **将来拡張**: フォーム認証の自動化、ヘッドレスブラウザでのログイン記録（フェーズ2でZAPと併せて）。

### 2.3 範囲外（明示する）
A04（設計欠陥）、ビジネスロジック、A01の権限境界網羅、A09（ログ監視）は**DASTでは原理的に取れない**。レポートに「ここから先は専門家（登録セキスペ等）へ」の導線を必ず置く。

## 3. 安全設計・ガードレール（機能より先に実装）

> 認証後スキャンは破壊的操作を踏むリスクがあるため、ガードレールを**最初に**入れる。

1. **ドメイン所有確認（必須）**:
   - **ローカル環境は自動スキップ**: `localhost`・`127.0.0.1`・`::1`・`*.local` は確認不要。開発・テスト（Juice Shop等）の障壁を排除。
   - **それ以外は確認必須**: 以下の2方式を選択可。スキャンはどちらかが通るまで実行不可。
     - **ファイル設置（優先）**: `/.well-known/goodast-verify/<token>` にファイルを置いてもらいHTTPアクセスで確認。初心者でも対応しやすい。
     - **DNS TXT（オプション）**: `goodast-verify=<token>` のTXTレコードをDNS設定（レジストラ/Cloudflare等）に追加 → DNSクエリで確認。エンジニア向けの代替手段。
2. **スコープ allowlist**: クロール／リクエストは登録ドメイン（+任意でサブパス）に限定。外部ドメインへの逸脱を禁止。
3. **危険パス除外（デフォルト）**: `logout`, `signout`, `delete`, `remove`, `destroy`, `admin/*` 等のパターンをデフォルト除外。認証後スキャンで誤ってログアウト／データ削除を踏まない。ユーザーが明示解除可能。
4. **保守的レート制限**: デフォルト低レート（例: 10–20 req/s、同時実行少なめ）。DoS化を防ぐ。UIで調整可。
5. **破壊的テストのオプトイン**: DoS系・状態変更を伴うテンプレートはデフォルト無効、明示有効化のみ。
6. **認証情報・APIキーの暗号化保存**: 平文ログ禁止。APIレスポンスで生キーを返さない（マスク表示）。
7. **テンプレート由来リスクへの配慮**: Nucleiバージョン固定、信頼できるテンプレートソースのみ。

## 4. アーキテクチャ

```
┌──────────────┐   HTTP/JSON    ┌─────────────────┐
│  Frontend     │ ◄───────────► │   API Server     │
│  (Nuxt 3)     │                │   (Go / Gin)     │
│  UI起点の全操作 │                │  受付/履歴/鍵管理  │
└──────────────┘                └────────┬────────┘
                                          │ enqueue job (river / PostgreSQL)
                                          ▼
                                ┌─────────────────┐
                                │  Scan Worker     │  ← Nuclei SDK をここだけに隔離
                                │  (Go, 別プロセス)  │     バージョン固定／GC配慮
                                │  非同期スキャン実行 │
                                └────────┬────────┘
                                          │ results (callback)
                                          ▼
                                    PostgreSQL

※ AI Layer（BYOK）はフェーズ2で追加。PoCはLLMなし。
```

### 設計上の肝
- **APIとスキャンワーカーを分離**: 公式が「Nucleiをサービス常駐させるのはセキュリティリスク」「SDKは破壊的変更あり（LTSなし）」と警告。Nuclei依存をワーカーに閉じ込め、影響を局所化する。
- **ジョブキューはPostgreSQLバックド**（例: `riverqueue/river`）。新規ミドルウェア（Redis等）を増やさず、既存スタックで完結。
- **AI呼び出しはAI層に集約**。ワーカーのスキャン結果（構造化済み）を入力に、LLMで平易化。スキャン処理とAI処理を分離してデバッグ容易に。

## 5. データモデル

サイト名を診断履歴の主キーにする（要件2）。

```sql
-- サイト（診断履歴のキー = name）
sites (
  id            uuid pk,
  name          text unique not null,   -- 例: "コーポレートサイト" → JSON target に使用
  base_url      text not null,
  ownership_verified  boolean default false,
  verify_method text,                    -- 'dns-txt' | 'file'
  verify_token  text,
  created_at    timestamptz
)

-- 認証プロファイル（セッション持ち込み、暗号化保存）
scan_credentials (
  id            uuid pk,
  site_id       uuid fk,
  auth_mode     text,                    -- 'none' | 'session'
  enc_headers   bytea,                   -- 暗号化した Cookie / Authorization
  created_at    timestamptz
)

-- 診断（1回のスキャン = 時系列の1点）
scans (
  id            uuid pk,
  site_id       uuid fk,
  status        text,                    -- queued|running|done|failed
  engine_version text,                   -- Nucleiバージョン記録
  started_at    timestamptz,
  finished_at   timestamptz,
  summary_json  jsonb                    -- schemaVersion 1.0 互換サマリ（後述）
)

-- 検出（個別findings）
findings (
  id            uuid pk,
  scan_id       uuid fk,
  template_id   text,
  title         text,
  severity      text,                    -- Critical|High|Medium|Low|Info
  url           text,                    -- 検出箇所（DASTなのでfile:lineでなくURL）
  cwe           text,
  remediation   text,
  status        text default 'open'      -- open|false_positive|fixed
  -- ai_explanation はフェーズ2で追加
)

-- ※ BYOK鍵テーブル（ai_keys）はフェーズ2で追加
```

### 5.1 Goodast Security Score（スコア設計）

スキャン結果を1つの数字に集約し、状態と遷移を初心者が直感的に読めるようにする。

**スコア計算式**（0〜100）
```
スコア = max(0, 100 − (Critical×40 + High×10 + Medium×3 + Low×1))
```

| 重大度 | 重み | 考え方 |
|---|---|---|
| Critical | 40 | 1件でスコアが致命的に下がる（即対応必須を数字で示す） |
| High | 10 | 3件で −30pt、放置すると赤に近づく |
| Medium | 3 | 積み重なって影響が出る |
| Low | 1 | 参考情報レベル |

**色分け**
| スコア | 色 | ラベル |
|---|---|---|
| 80〜100 | 🟢 緑 | 良好 |
| 60〜79 | 🟡 黄 | 要注意 |
| 40〜59 | 🟠 橙 | 危険 |
| 0〜39 | 🔴 赤 | 危機 |

> 重み値は実装後の実データで調整する余地あり。Criticalが40はかなり大きい設計（1件で赤圏内に入る）だが、「Criticalが残っていたら赤」という意図が初心者にも伝わりやすいため採用。

### 5.2 スキャンサマリ（ダッシュボード描画用の内部データ）

`scans.summary_json` に格納し、ダッシュボードの「状態」と「遷移」を描画するためのデータ。
**ツール内で完結させる。** 将来のスキル統合を阻害しないよう構造はシンプルに保つ。

```json
{
  "date": "2026-06-27",
  "target": "コーポレートサイト",
  "scope": "https://example.com",
  "authMode": "session",
  "score": 67,
  "scoreLabel": "要注意",
  "findings": {
    "critical": 0, "high": 1, "medium": 3, "low": 2,
    "total": 6, "falsePositives": 0
  },
  "topFindings": [
    { "id": "V-01", "title": "格納型XSSの可能性", "severity": "High", "location": "https://example.com/profile" }
  ]
}
```

> **メモ**: フィールド名やtypeは実装時に最適化してよい。既存スキル（dev-skills）との整合は後続フェーズで「より良い統合」として別途設計するため、本PoCではそこに縛られない。状態と遷移が描ければ形式は自由。

## 6. 画面 / UIフロー（UI起点）

1. **サイト登録**: サイト名 + ベースURL入力 → ドメイン所有確認（DNS TXT/ファイル）ガイド表示 → 確認完了で有効化。
2. **スキャン設定ウィザード**:
   - プリセット選択（軽量 / 標準 / 詳細）= 安全なデフォルト。
   - 認証: 「なし」 or 「ログイン状態を持ち込む」（Cookie/Bearer貼り付けガイド付き）。
   - 危険パス除外の確認（デフォルトON）。
3. **スキャン実行**: 非同期。進捗をリアルタイム表示（queued→running→done）。
4. **結果レポート**: 重大度カラー、CVSS/CWE、検出URL、修正方法。AI平易化トグル（BYOK設定済みなら）。末尾に「専門家への相談」導線。
5. **診断履歴**: サイト名ごとに過去スキャン一覧。
6. **ダッシュボード（本PoCの中心）**: サイト単位で2層構成。
   - **上段 — 状態（今どうか）**: Goodast Security Scoreを大きな数字＋色（🟢🟡🟠🔴）で表示。前回との差分（例: `+5↑` / `-12↓`）を添える。重大度別カウントのサマリカードを並べる。
   - **中段 — 遷移（どう変わったか）**: スコアの時系列折れ線チャート。改善/悪化のトレンドが一目でわかる（遷移の主役）。
   - **下段 — 内訳（なぜそのスコアか）**: 重大度別の積み上げ棒チャート。どのレベルが増減したかを補足説明。
   - Chart.js、ダークテーマ。2回以上のスキャンで遷移が描画される。
7. **設定**: スキャンプリセット（軽量/標準/詳細）、危険パス除外の編集。BYOK設定はフェーズ2で追加。

## 7. AI活用（BYOK）

- **PoCでの用途 = レポート平易化のみ**（最小）。各findingを「何が・なぜ危険か・どう直すか」を初心者向けの言葉に変換。
- **BYOK実装定石**:
  - プロバイダ選択 + OpenAI互換エンドポイント対応（ローカルLLMも可）。
  - キーは暗号化保存・平文ログ禁止・接続テストあり・直課金（ツール側マークアップなし）。
  - トークン消費の可視化（コスト透明性）。
- **AIに約束させないこと**: 検出の網羅性・正確性。AIは補助。自律探索（最大13%水準）は前面に出さない。
- **後続**: 設定ウィザードの自然言語化、トリアージ（誤検知判定）、探索高度化。

## 10. 完了条件（PoC DoD）

### 機能確認
- [ ] UIからサイト名+URLを登録し、ドメイン所有確認を経てスキャンを開始できる。
- [ ] 認証なしページ・**ログイン後ページ（セッション持ち込み）**の双方をスキャンできる。
- [ ] スキャンが非同期実行され、進捗がUIに反映される（API/Worker分離が機能）。
- [ ] 危険パス除外・レート制限・allowlistが効いている（ログアウト等を踏まない）。
- [ ] 結果がレポート表示される（重大度・検出URL・修正方法）。末尾に「専門家への相談」導線あり。
- [ ] サイト名で履歴一覧が見られる。
- [ ] **直近の検査状態（Goodast Security Score＋色＋前回差分＋重大度別カウント）が可視化される。**
- [ ] **2回以上のスキャンで時系列のスコア折れ線＋重大度別積み上げ棒が描画され、遷移（改善/悪化）が読み取れる。**（本PoCの中心要件）

### 検知精度の検証

テスト環境として **OWASP Juice Shop**（Docker）を使用する。現代的なSPA構成・REST API・認証機能を備え、Nucleiの公式テンプレートも対応済み。goodastのPoC要件（認証後スキャン）と直接対応する。

**検証フロー**

```
1. Juice Shop を Docker で起動
2. Nuclei CLI で直接スキャン → 結果をベースライン（正解）とする
3. goodast UI から同条件でスキャン → 結果を比較
   ├─ 一致         → ラッパー実装が正常
   ├─ goodast側が欠落 → SDK統合のバグ（要修正）
   └─ goodast側が多い → AI誤検知判定の問題（要確認）
4. 認証後スキャン検証
   ├─ Juice Shop にブラウザでログイン → CookieをgoodasのUIに貼り付け
   └─ 未認証スキャンとの差分でカバレッジ拡大を確認
```

**重大度の判定ソースについて**
- **PoC**: Nucleiテンプレートの `severity` フィールドをそのまま重み計算に使用（テンプレート作者の判定はおおむねCVSSに準拠）。
- **CVSSv3.1 → 重大度マッピング（NVD標準）**:
  - 9.0〜10.0 → Critical / 7.0〜8.9 → High / 4.0〜6.9 → Medium / 0.1〜3.9 → Low
- **将来（フェーズ2）**: CVE紐づき検出はNVDからCVSSスコアを取得して上書き。CVSS値をレポートに表示し、専門家への説明資料としても使えるように。

**完了基準**
- [ ] Nuclei CLIとgoodasのfinding件数・severityが一致している（欠落ゼロ）。
- [ ] 認証後スキャンで未認証時より多くのfindingが検出される（カバレッジ拡大の確認）。
- [ ] 「初回スキャン完了まで5分以内」を初心者が達成できる（UX検証）。

## 12. 決定事項・残確認事項

**決定済み**
- ジョブキュー: `river`（PostgreSQLバックド）
- 暗号化方式: **アプリケーションレイヤー**（DB依存を避け、テスト・移植が容易）
- ドメイン所有確認の目的: **他人のサイトへの不正スキャンを防ぐ安全装置**。「自分が管理するサイトしか検査できない」を強制する。法的・倫理的にも必須。
- ドメイン所有確認の方式:
  - **ファイル設置を優先**（`/.well-known/goodast-verify/<token>` にファイルを置いてもらいHTTPアクセスで確認）。初心者でもサーバにアクセスできれば対応可能。
  - **DNS TXT はオプション**（`goodast-verify=<token>` のTXTレコードをDNS設定に追加 → DNSクエリで確認）。エンジニア向けの代替手段。
  - **ローカル環境は所有確認をスキップ**: `localhost`、`127.0.0.1`、`::1`、`*.local` は確認不要。開発・テスト時の障壁を排除し、Juice Shopでの検証も摩擦なく行える。

- **Nucleiテンプレート配布**: バージョン固定＋起動時自動ダウンロード。テンプレート自体はリポジトリに含めない（容量・ライセンス管理の観点）。Dockerイメージビルド時またはワーカー起動時に `nuclei-templates` の特定バージョンをGitHubから取得。バージョンはコンフィグで明示固定し、予期しないテンプレート更新による挙動変化を防ぐ。

**残確認事項**
- なし。全項目確定。Claude Codeへの実装指示に進める状態。
