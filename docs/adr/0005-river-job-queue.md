# ADR-0005: ジョブキューにriverを採用する

Status: Accepted
Date: 2026-06-27

## Context

スキャンは数分かかる非同期処理であり、APIとワーカーを疎結合に繋ぐジョブキューが必要（ADR-0001）。選択肢を比較した。

| 選択肢 | 追加インフラ | 備考 |
|---|---|---|
| `riverqueue/river` | なし（PostgreSQL利用） | Goネイティブ、リトライ・優先度・スケジュール標準装備、本番でも使える |
| 自前ポーリング（`SELECT FOR UPDATE SKIP LOCKED`） | なし | 最小実装だが機能を自作する必要あり |
| `asynq` | Redis | ブローカーが増える |

## Decision

**`riverqueue/river`** を採用する。

- PostgreSQLをバックエンドにするため、既存スタック（Go + PostgreSQL）に追加インフラなしで乗る。
- リトライ・優先度・スケジュール実行などスキャンジョブ管理に必要な機能が標準で揃う。
- 自前ポーリングより堅牢で、本番化までそのまま使える。

## Consequences

- **利点**: Redis等のブローカーを増やさず構成がシンプル。スキャンジョブのライフサイクル管理（リトライ・失敗ハンドリング）を自作せずに済む。
- **トレードオフ**: riverのスキーマ・マイグレーションがDBに追加される。→ アプリのマイグレーション管理に統合する。
- PostgreSQLがジョブとアプリデータの双方を担うため、負荷増大時はDBがボトルネックになりうる。PoC規模では問題にならない。本番スケール時に再評価する。
