# ADR-0001: APIとスキャンワーカーを分離する

Status: Accepted
Date: 2026-06-27

## Context

goodast は Nuclei v3 Go SDK をエンジンに使う。Nuclei公式ドキュメントは2点を警告している。

1. 「Nucleiをサービスとして常駐させるのはセキュリティリスクがある」
2. 「SDKは活発に開発中で破壊的変更がありうる（LTS・stable版なし）」

加えてスキャンは数分〜十数分かかり、高リソース・潜在的に破壊的（認証後スキャンで状態変更を踏む可能性）。これをAPIサーバプロセスに同居させると、APIの応答性・安定性・セキュリティ境界が損なわれる。

## Decision

APIサーバ（`api/`）とスキャンワーカー（`worker/`）を**別プロセス**に分離する。

- Nuclei SDK への依存は `worker/` にのみ置く。`api/` からは一切importしない。
- 両者は `river`（PostgreSQLバックドのジョブキュー）経由で疎結合に連携する。
- ワーカーはNucleiのバージョンを固定して実行する（ADR-0002）。

## Consequences

- **利点**: Nuclei依存と破壊的変更の影響をワーカーに局所化できる。スキャンの負荷・クラッシュがAPIに波及しない。セキュリティ境界が明確になる。スケール時はワーカーのみ水平展開できる。
- **トレードオフ**: プロセス間連携（ジョブキュー）が必要になり、構成がやや複雑化する。→ `river` でPostgreSQL内に閉じることで追加インフラは増やさない（ADR-0005）。
- ワーカーのGC・メモリ挙動はNucleiの推奨（低メモリ制限を課さない等）に従う。
